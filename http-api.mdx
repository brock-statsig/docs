<Warning title="Prefer official SDKs">
Use the HTTP API only when an official Statsig SDK is unavailable for your platform or you need a custom integration. SDKs provide better performance, built-in error handling, and idiomatic interfaces.
</Warning>

Follow these steps to authenticate and send requests:

1. [Create a Statsig account](#step-1-create-an-account)
2. [Generate an API key](#step-2-get-an-api-key)
3. [Issue HTTPS requests](#step-3-issue-api-requests)

---

## Step 1 – Create an account

Create a free account on the [Statsig sign-up page](https://console.statsig.com/sign_up). The console lets you manage Feature Gates, Dynamic Configs, Experiments, and collaborate with teammates.

---

## Step 2 – Get an API key

Navigate to the [Statsig Console](https://console.statsig.com/) and create the key you need:

- **Server-side secret key** – Use on secure backend services only.
- **Client SDK key** – Safe to embed in mobile or client applications.

<Tip>
When handling sensitive logic or data server-side, prefer the server-side secret key. Use the client SDK key for public-facing code.
</Tip>

---

## Step 3 – Issue API requests

All endpoints are HTTPS `POST` requests authenticated with the `statsig-api-key` header. Payloads are JSON objects in the request body. Statsig automatically logs exposure events so downstream metrics stay accurate.

### StatsigUser schema

Every call requires user context. Supply at least one identifier and any additional targeting fields you rely on.

```ts
interface StatsigUser {
  userID?: string; // Provide at least one identifier when possible
  email?: string;
  ip?: string;
  userAgent?: string;
  country?: string;
  locale?: string;
  appVersion?: string;
  custom?: Record<string, string | number | boolean | string[]>;
  privateAttributes?: Record<string, string | number | boolean | string[]>;
  customIDs?: Record<string, string>;
  statsigEnvironment?: { tier: string };
}
```

## Log events

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --header "STATSIG-CLIENT-TIME: <local_time>" \
  --request POST \
  --data '{"events": [{"user": { "userID": "42" }, "time": 1616826986211, "eventName": "test_api_event"}]}' \
  "https://events.statsigapi.net/v1/log_event"
```

<Note>
`STATSIG-CLIENT-TIME` normalizes timestamps across devices. Without it, client clocks can skew event timing. See the [C++ SDK example](https://github.com/statsig-io/cpp-client-sdk/blob/a62df973388a56669de2a05f0630a65570a775bd/src/statsig/internal/network_service.hpp#L148).
</Note>

#### Event payload

```ts
interface LogEventsRequest {
  events: StatsigEvent[];
}

interface StatsigEvent {
  eventName: string;
  value?: number | string;
  time: number; // Unix timestamp
  user: StatsigUser;
  metadata?: Record<string, string>;
}
```

Success response:

```json
{"success": true}
```

### Log an event with a custom environment

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --header "STATSIG-CLIENT-TIME: <local_time>" \
  --request POST \
  --data '{"events": [{"user": { "userID": "42", "statsigEnvironment": {"tier": "staging"} }, "time": 1616826986211, "eventName": "test_api_event"}]}' \
  "https://events.statsigapi.net/v1/log_event"
```

## Feature gates

### Check a gate

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{"user": { "userID": "42" }, "gateName": "<YOUR-GATE-NAME>"}' \
  "https://api.statsig.com/v1/check_gate"
```

### Check multiple gates

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{"user": { "userID": "42" }, "gateNames": ["<YOUR-GATE-NAME-1>", "<YOUR-GATE-NAME-2>"]}' \
  "https://api.statsig.com/v1/check_gate"
```

Sample response:

```json
[
  {"name": "YOUR-GATE-NAME-1", "value": false, "rule_id": "123", "group_name": "group123"},
  {"name": "YOUR-GATE-NAME-2", "value": false, "rule_id": "123", "group_name": "group123"}
]
```

## Dynamic configs and experiments

### Get a dynamic config

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{"user": { "userID": "42" }, "configName": "<YOUR-CONFIG-NAME>"}' \
  "https://api.statsig.com/v1/get_config"
```

Response:

```json
{"name": "YOUR-CONFIG-NAME", "value": {"a": 1, "b": 2}, "group": "123", "rule_id": "123", "group_name": "group123"}
```

### Fetch an experiment config

Same endpoint as dynamic configs; Statsig logs the correct exposure automatically.

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{"user": { "userID": "42" }, "configName": "<YOUR-EXPERIMENT-NAME>"}' \
  "https://api.statsig.com/v1/get_config"
```

Response:

```json
{"name": "YOUR-EXPERIMENT-NAME", "value": {"color": "blue", "shape": "circle"}, "group": "123", "rule_id": "123", "group_name": "group123"}
```

## Custom exposure logging

Use custom exposure logging when you manage bucketing outside Statsig or need to attribute exposures manually.

### Experiment exposure

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{"exposures": [{"user": {"userID": "user_id_12345"}, "experimentName": "analytics_only_experiment", "group": "Daily Deals"}]}' \
  "https://events.statsigapi.net/v1/log_custom_exposure"
```

### Feature gate exposure

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{"exposures": [{"user": {"userID": "user_id_12345"}, "gateName": "saleBanner", "group": "Controls Access", "passes": true}]}' \
  "https://events.statsigapi.net/v1/log_custom_exposure"
```

### Secondary exposures

Secondary exposures capture holdouts or targeting gates evaluated alongside the primary exposure.

```bash
curl \
  --header "statsig-api-key: <YOUR-SDK-KEY>" \
  --header "Content-Type: application/json" \
  --request POST \
  --data '{
  "exposures": [
    {
      "user": { "userID": "user_id_12345" },
      "gateName": "saleBanner",
      "group": "Controls Access",
      "passes": true,
      "secondaryExposures": [
        {
          "gate": "test_holdout",
          "gateValue": "true",
          "ruleID": "3CQWjvi4HXcHsGIuuffKoe:10.00:2"
        }
      ]
    }
  ]
}' \
  "https://events.statsigapi.net/v1/log_custom_exposure"
```
